    平滑更新部署（滚动更新）。相当于阿里云的容器服务里的触发器，触发更新。

    kubectl 创建https secret

    基于header头的ingress流量转发

    通过设置ingress配置的注解：匹配请求的header（service-match），version: 3.6.* 则转发值service: abroadapitestservice，否则转发到leshuapiservice。在此基础上还可以设置权重（service-weight），在包含匹配header头的请求中，一定比例的请求转发至两个版本的service。

    apiVersion: extensions/v1beta1
kind: Ingress
metadata:
  name: leshuingress
  namespace: test
  annotations:
    kubernetes.io/ingress.class: "nginx"
    nginx.ingress.kubernetes.io/service-match: |
        abroadapitestservice: header("version", /^3.6.*$/)
    nginx.ingress.kubernetes.io/service-weight: |
        abroadapitestservice: 50, leshuapiservice: 50
spec:
  rules:
  - host: hk.ad6755.com
    http:
      paths:
      - path:
        backend:
          serviceName: leshuapiservice
          servicePort: 80
      - path:
        backend:
          serviceName: abroadapitestservice
          servicePort: 80
  tls:
  - hosts:
    - hk.ad6755.com
    secretName: leshuingress-secret0

    原理：创建ExternalName类型的service时会向coredns组件注册一条cname类型的记录：test-service.default — rm-bp1768cgx1e2an6w0.mysql.rds.aliyuncs.com

    关于coredns：之前公司有爬虫业务，爬取数据时经常会报域名解析失败之类的错误，后来得知，在容器里解析域名时首先会向coredns服务查询，如果没查到，则会向宿主机设置的dns服务器查询。爬虫业务频繁使用coredns服务，导致cordns不断重启，因此有了业务上的报错。所以一般情况下，如果容器不需要使用service调用服务时，建议将容器的dnsPolicy设置为default。

    优雅终止pod
    生产中遇见的问题：

    web服务，由ingress代理到pod。ingress日志报了502，对比时间节点，502的时刻刚好是一次滚动更新，据此得知：ingress将流量代理到了已经被删除的pod（对比pod ip地址）。

    pod更新过程：

    启动新pod => 等待新pod进入RUNNING状态 => 创建endpoint，将新服务纳入负载均衡 => 设置旧pod为Terminating状态 => preStop Hook执行 & 从负载均衡器移除旧pod的endpoint => 等待preStop Hook执行结束 => 给旧pod中的容器一号进程发送一个sigterm信号 => 等待旧pod终止，如果超过了terminationGracePeriodSeconds所设置的时间则发送sigkill信号，并删除pod。参考

    实验：

    给pod增加以下配置

    lifecycle:
  preStop:
    exec:
      command:
      - sleep
      - "30"

    在增加prestop hook之后，手动触发pod滚动更新deployment，新pod进入running之后，旧pod进入Terminating状态的同时容器内部开始执行sleep（进入容器执行了一个监控脚本，监控是否有sleep进程），同时修改负载均衡器的endpoint（移除旧pod的endpoint）。

    分析出现502的原因：

    旧pod首先被设置成Terminating状态，由于没有设置pre stop hook，这一步直接跳过，同时容器收到sigterm信号，容器内部的服务停止了。但是旧pod的endpoint并没有及时的从负载均衡后端移除（发送sigterm信号和修改负载均衡器的endpoint是同步执行的），流量依然被转发至旧pod，而旧pod中的服务已经终止，so 导致了502。

    解决
    lifecycle:
  preStop:
    exec:
      command:
      - sleep
      - "30"
terminationGracePeriodSeconds： 60

    增加preStop Hook —- 回调，在pod被设置为terminating状态之后，不要立马发送sigterm信号，而是执行preStop Hook，我这里执行sleep 30，也就是在pod被设置为Terminating状态之后什么也不做，等待30秒，里面的服务依旧运行，给负载均衡更新endpoint留时间，30s之后负载均衡器的endpoint必然更新完了，此时再给pod发送终止信号，服务优雅停止，更新。

    pod调度
    deployment 或者 Rc 全自动调度

    根据k8s自己的算法，遍历节点，挑选合适的节点去调度

    NodeSelector：定向调度

    给Node打标签，定义pod时使用NodeSelector字段进行定向调度

    NodeAffinity：Node亲和性调度
    1. RequiredDuringSchedulingIgnoreDuringExection —- 硬限制，等同于NodeSelector
    2. PreferedDuringSchedulingIgnoreDuringExection —– 软限制，会尝试但不强求

    IgnoreDuringExection：如果一个pod所在的节点在pod运行期间标签发生了变更，不在符合该pod的节点亲和性需求，则系统忽略Node上的Label变化，该pod继续在该Node上运行

    PodAffinity：Pod亲和与互斥调度
    Taints和Toletations

    与NodeAffinity相似，不过作用相反：Traint要求Node拒绝pod的运行，是定义在pod上的属性。

    Pod优先级调度

    给pod设置优先级，数字越大优先级越高。

    pod中挂载单个文件

    在pod中使用volume挂载配置文件的时候，如果原目录下存在其它文件，挂载后将被覆盖掉。某些时候我们的需求是将单个文件挂载到容器里的某个目录，但是不要影响目录下的其它文件，例如：nginx配置文件挂载。可以使用subPath解决此问题。
